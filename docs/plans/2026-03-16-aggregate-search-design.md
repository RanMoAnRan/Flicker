# 浮光全片源聚合搜索设计

日期：2026-03-16

## 背景

当前首页搜索仅会查询“当前选中的片源”。用户希望输入一个关键词后，能一次性搜索所有已配置片源的内容，而不是逐个切换片源重复搜索。

现有项目已经支持两类接口：

- `vod` 影视播放源
- `art` 资讯内容源

因此聚合搜索设计必须同时兼容两类源，并保证搜索结果进入详情页时仍能命中正确的数据源。

## 目标

- 首页搜索时并发查询所有已配置片源
- 将全部结果聚合成一个统一列表展示
- 每条结果明确标注来源片源
- 点击结果进入详情页时，始终使用该结果所属片源加载详情
- `vod` 结果继续可播放，`art` 结果继续走阅读详情

## 非目标

- 不做后端聚合搜索接口
- 不做复杂智能去重
- 不做跨全部片源的深页分页
- 不做按片源二次筛选器

## 方案选择

### 方案一：前端并发搜索全部片源并聚合展示

前端读取所有数据源，对每个源并发发起搜索请求，统一走已有的数据适配层，将结果合并、轻度去重、统一排序后展示。

优点：

- 与当前项目结构最契合
- 开发成本低，见效快
- 适合当前本机自用场景

缺点：

- 片源较多时，搜索耗时会高于单源搜索

### 方案二：前端逐个片源搜索并流式展示

先返回最快的片源结果，再逐步把其他片源结果插入页面。

优点：

- 首屏更快看到结果

缺点：

- 结果顺序会持续变化
- 状态管理复杂
- 体验不够稳定

### 方案三：代理层新增后端聚合接口

由本地代理统一并发查询全部片源，再将结果一次性返回给前端。

优点：

- 长期扩展性最好
- 后续缓存、排序、去重更可控

缺点：

- 当前阶段复杂度偏高
- 会显著增加服务端逻辑

### 结论

采用方案一。原因是它能够在不显著增加系统复杂度的前提下，完整满足“搜索所有片源”的需求，同时复用现有前端数据适配层。

## 数据模型设计

聚合搜索结果中的每一项，在现有标准化模型基础上增加以下字段：

- `source_id`
- `source_name`
- `source_kind`
- `source_url`

其中：

- `source_kind` 取值为 `vod` 或 `art`
- `source_id` 对应本地已配置片源 ID

这些字段用于：

- 在卡片上展示来源片源
- 在详情页和播放器页中还原正确片源上下文

## 页面行为设计

### 首页

首页新增“聚合搜索模式”概念。

进入方式：

- 用户输入关键词并执行搜索

表现：

- 并发搜索所有已配置片源
- 标题改为 `搜索：关键词`
- 页面摘要显示：
  - 已搜索片源数量
  - 成功片源数
  - 失败片源数
  - 聚合结果总数
- 结果卡片显示来源片源标签

聚合模式下的限制：

- 分类栏不再作为跨片源分类过滤器使用
- 顶部片源选择器保留，但只影响“默认浏览片源”，不影响当前聚合搜索结果
- 页面上应提示当前为全片源搜索结果

退出方式：

- 清空搜索框后恢复单片源浏览模式

### 详情页

详情页优先读取 URL 中的 `source_id`。

示例：

- `detail.html?id=123&source_id=4`

行为：

- 若 URL 中有 `source_id`，则从该片源加载详情
- 若没有，再退回当前默认片源

这样可以保证聚合搜索点开的内容不会误用错误片源。

### 播放页

播放页同样优先读取 `source_id`。

行为：

- `vod` 条目按所属片源继续播放
- `art` 条目继续显示“资讯源不支持播放，请返回详情阅读”

## 排序与去重策略

### 排序

默认按以下顺序排序：

1. 更新时间倒序
2. 有封面优先
3. 有摘要优先
4. 保留原始返回顺序作为兜底

### 去重

第一版仅做轻去重：

- 仅在同一片源内部，对“同名且同时间”的条目去重
- 不跨片源强行去重

原因：

- 不同片源可能存在同名不同资源
- `vod` 与 `art` 混合搜索时，跨源强去重误判风险高

## 异常处理

- 单个片源搜索失败，不影响其他片源结果展示
- 仅当全部片源都失败时，显示整体错误态
- 若部分片源失败，则正常展示已有结果，并提示失败片源数量

## 实现范围

### `js/app.js`

- 增加根据 `source_id` 获取片源的方法
- 增加聚合搜索方法
- 为标准化条目补齐来源字段
- 增加聚合排序与轻去重能力
- 更新跳转方法，使详情和播放链接可携带 `source_id`

### `index.html`

- 将搜索改为并发查询全部片源
- 增加聚合搜索模式状态
- 调整标题、摘要、提示文案
- 在结果卡片显示来源片源

### `detail.html`

- 优先根据 `source_id` 加载对应片源详情

### `player.html`

- 优先根据 `source_id` 加载对应片源播放信息

### `css/style.css`

- 如有必要，补充来源片源标签样式

## 验证计划

- 检查 `js/app.js` 语法
- 检查页面内联脚本语法
- 使用至少两个片源执行同一关键词搜索
- 验证聚合结果中能看到来源片源标签
- 随机点开多个聚合结果，确认详情使用的是正确片源
- 对 `vod` 结果验证可进入播放流程
- 对 `art` 结果验证进入详情阅读，不误入播放器

## 风险

- 片源数量增加后，前端并发搜索响应时间会变长
- 某些片源返回时间格式不规范，可能影响排序
- 不跨源去重会保留一部分重复内容，但能避免误杀

## 后续可扩展方向

- 聚合搜索结果增加“仅看某片源”筛选
- 聚合搜索结果增加按 `vod/art` 类型过滤
- 代理层增加缓存和统一聚合接口
- 引入更精细的跨源去重逻辑
