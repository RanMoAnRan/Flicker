# Flicker 安卓 App 设计

## 目标

在保留当前 Web 原型站点的前提下，为项目规划一个长期可维护的安卓 App 方案，覆盖影视与音乐双模块，并尽量将核心能力收敛到客户端本地实现。

本方案面向以下约束：

- 保留现有仓库中的 Web 版本，不与安卓工程混放。
- 安卓 App 仅考虑自用或小范围安装，不以上架应用市场为前提。
- 第一阶段目标是做出可长期演进的正式 App 基线，而不是简单 WebView 包壳。

## 现状

- 当前前端为原生 HTML、CSS、JavaScript，多页面结构，核心逻辑集中在 `js/app.js` 与 `js/music-library.js`。
- 当前服务端为 `proxy-server.js`，负责静态资源分发、`/api/proxy` 转发，以及 `/api/music/*` 音乐相关接口。
- 影视部分依赖多片源聚合与字段适配。
- 音乐部分依赖插件目录、插件搜索、播放地址获取、歌词获取与播放队列。

现有 Node 服务中的部分职责，本质上是为浏览器环境补齐跨域、代理和插件执行能力；进入原生安卓环境后，这些职责需要重新拆到客户端架构中。

## 方案选择

候选方向有三类：

1. WebView 包壳：开发快，但长期会积累较重技术债，不适合作为最终架构。
2. Flutter 重做：适合未来强烈需要跨平台时考虑，但当前以安卓单平台为目标，且核心播放与插件能力仍需原生兜底。
3. Kotlin 原生 Android：最符合长期产品化目标，能够更好承载播放、后台服务、通知、锁屏控制、插件运行与本地能力沉淀。

最终选择：采用 `Kotlin + Jetpack Compose` 的原生安卓方案作为目标架构。

实施节奏上允许借鉴“渐进迁移”的思路，但最终形态不以 WebView 作为主架构。

## 仓库结构设计

安卓工程必须独立放在顶级目录 `flicker-app/` 中，现有 Web/Node 项目继续保留在仓库根目录。

建议结构如下：

```text
Flicker/
├── index.html
├── detail.html
├── music.html
├── player.html
├── settings.html
├── js/
├── css/
├── proxy-server.js
├── docs/
│   └── plans/
└── flicker-app/
    ├── settings.gradle.kts
    ├── build.gradle.kts
    ├── gradle.properties
    ├── app/
    ├── core-common/
    ├── core-network/
    ├── core-database/
    ├── core-player/
    ├── core-plugin/
    ├── feature-video/
    ├── feature-music/
    └── feature-settings/
```

采用该结构的原因：

- 现有 Web 原型可继续运行，不被安卓工程污染。
- 安卓代码、Gradle 配置和模块边界集中在 `flicker-app/` 中，后续维护清晰。
- 便于以当前 Web 实现作为业务参考，同时逐步完成安卓端原生重建。

## 技术栈设计

- 语言：`Kotlin`
- UI：`Jetpack Compose`
- 架构：`MVVM` 或 `MVI`
- 网络：`OkHttp`
- 序列化：`kotlinx.serialization`
- 数据库：`Room`
- 偏好存储：`DataStore`
- 图片加载：`Coil`
- 视频播放：`Media3 ExoPlayer`
- 音乐播放：`Media3 + MediaSessionService`
- 后台任务：`WorkManager`
- JS 插件运行时：`QuickJS` 或同类轻量 JavaScript 引擎

## 总体架构

安卓 App 采用分层架构，避免页面直接持有网络、解析和播放逻辑。

### Presentation

负责 Compose 页面、导航与状态渲染，覆盖：

- 首页
- 影视详情页
- 视频播放页
- 音乐库页
- 音乐播放控制界面
- 设置页

### Domain

负责核心业务用例，例如：

- 搜索影视
- 读取分类列表
- 获取详情
- 解析播放线路
- 搜索音乐
- 获取歌词
- 切换上一首与下一首

### Data

负责远程数据、本地缓存和用户数据持久化：

- 片源配置
- 当前片源
- 插件目录元数据
- 历史记录
- 收藏记录
- 搜索历史

### Player

单独抽离音视频播放能力：

- 视频播放链路
- 音乐后台播放
- 通知栏控制
- 锁屏控制
- 播放队列
- 进度恢复

### Plugin

独立的音乐插件运行层，负责：

- 加载插件目录
- 执行插件脚本
- 管理插件超时与异常隔离
- 暴露受控桥接接口

### Network

替代当前 `/api/proxy` 的职责：

- 直接请求上游接口
- 注入请求头
- 错误码归一化
- 处理重定向、超时与解析异常

## 模块划分

建议从一开始按多模块组织：

- `app`：应用入口、导航、依赖注入
- `feature-video`：影视首页、搜索、详情、播放入口
- `feature-music`：音乐搜索、歌词、播放队列、推荐榜单
- `feature-settings`：片源管理、主题、调试项
- `core-common`：通用工具、结果封装、日志、时间与格式化能力
- `core-network`：HTTP 请求、错误映射、解析工具
- `core-database`：Room、DAO、实体模型
- `core-player`：视频/音频播放器能力与服务
- `core-plugin`：音乐插件运行时与桥接

## 现有项目到安卓模块的映射

- `js/app.js` 对应未来的 `feature-video`、`feature-settings` 与部分 `core-common`
- `js/music-library.js` 对应未来的 `feature-music`
- `proxy-server.js` 中的静态资源分发职责在安卓端消失
- `proxy-server.js` 中的代理、错误映射、插件桥接职责拆分到 `core-network` 与 `core-plugin`

迁移重点不是复用现有前端代码本身，而是复用：

- 片源模型理解
- 接口字段适配经验
- 播放线路解析规则
- 音乐插件结构和兼容策略
- 错误处理经验

## 影视链路设计

影视模块采用“片源配置驱动”的统一数据流：

1. 读取当前启用片源。
2. 由 `VideoRepository` 组织请求。
3. 通过 `SourceAdapter` 处理不同源的字段差异。
4. 将结果映射为统一模型后返回界面层。

建议统一模型：

- `VideoSource`
- `VideoCategory`
- `VideoListItem`
- `VideoDetail`
- `Episode`
- `PlayLine`
- `ArticleDetail`

视频播放采用“两段式处理”：

1. 详情页阶段拿到线路和剧集信息。
2. 进入播放页后再解析最终可播地址。

这样可以减少详情页阻塞，并将播放异常单独收口在解析器与播放器层。

## 音乐链路设计

音乐模块采用“插件驱动 + 统一模型”的方式实现：

1. App 启动时加载插件目录与缓存元数据。
2. `PluginManager` 返回可用插件清单。
3. 搜索时并发调度多个插件执行搜索能力。
4. 将结果映射为统一音乐模型。
5. 播放时调用对应插件获取播放地址。
6. 歌词异步获取并与播放进度绑定。

建议统一模型：

- `MusicPlugin`
- `Song`
- `Album`
- `Artist`
- `LyricLine`
- `PlaylistRef`
- `MediaStream`

音乐模块的开发难度显著高于影视模块，原因在于其不仅是页面功能，还包括：

- 插件运行时
- 多插件聚合
- 歌词同步
- 后台播放
- 通知与锁屏控制
- 播放队列与上下首逻辑

## 插件运行层设计

插件层应采用受控沙箱，不允许任意脚本直接访问安卓系统能力。

建议设计：

- 使用 `QuickJS` 创建独立插件上下文。
- 每个插件单独执行，彼此隔离。
- 通过 Kotlin 向插件暴露有限桥接接口。
- 所有网络能力统一走 `HttpBridge`。

建议暴露的桥接能力：

- `http.get`
- `http.post`
- `crypto`
- `time`
- `storage`
- `log`
- `parseHtml`
- `parseJson`

必须具备三类保护：

- 超时控制：搜索、获取播放地址、获取歌词必须设定时限。
- 异常隔离：单插件失败不得拖垮整体搜索或播放流程。
- 兼容策略层：保留当前项目中插件兼容映射思路，但集中管理，避免散落在业务代码中。

## 错误处理设计

从一开始对错误进行分层建模：

- 网络错误
- 上游源错误
- 解析错误
- 播放错误
- 插件执行错误

建议统一为内部错误模型，例如：

- `AppError.Network`
- `AppError.SourceUnavailable`
- `AppError.ParseFailed`
- `AppError.PlaybackFailed`
- `AppError.PluginTimeout`
- `AppError.PluginCrashed`

页面交互策略：

- 首页和搜索页局部失败，不全页白屏。
- 详情页允许重试并保留已获取的基础信息。
- 播放页优先尝试自动切换线路。
- 音乐聚合搜索中单插件失败不影响其他插件结果。
- 后台播放失败时允许跳过当前项并提示。

## 缓存与离线边界

缓存分三层：

### 轻缓存

- 搜索历史
- 播放历史
- 收藏记录
- 当前片源和插件偏好

### 中等缓存

- 海报与封面图
- 歌词
- 已解析的播放线路短时缓存
- 插件目录与元数据

### 重缓存

- 视频离线下载
- 音乐文件完整缓存

第一阶段不做重缓存能力，以免过早引入下载、空间管理、失效清理和复杂文件生命周期问题。

离线边界定义：

离线时可用：

- 查看历史记录
- 查看收藏
- 查看缓存过的封面和部分详情摘要
- 查看已缓存歌词
- 查看本地保存的片源和插件配置

离线时不可用：

- 在线搜索
- 在线播放
- 实时歌词刷新
- 动态榜单和插件动态内容

“能力本地化”在本方案中指关键逻辑和运行时在 App 内，而非承诺完全离线可用。

## MVP 范围

第一阶段只覆盖以下能力：

- 影视首页浏览
- 影视搜索
- 影视详情
- 选集播放
- 音乐搜索
- 音乐播放
- 歌词展示
- 上一首、下一首
- 片源管理
- 主题切换
- 播放历史
- 收藏
- 后台音乐播放
- 通知栏控制
- 锁屏控制

第一阶段明确不做：

- 离线下载
- 云同步
- 账号体系
- 推荐系统
- 平板专项适配
- 插件市场完整产品化改造

## 实施顺序

建议按以下顺序推进：

1. 搭建 `flicker-app/` 安卓工程基线和多模块结构。
2. 优先完成影视主链路：首页、搜索、详情、播放。
3. 完成音乐基础链路：单插件搜索、播放、歌词。
4. 扩展音乐聚合能力：多插件搜索、播放队列、上下首。
5. 补齐设置、历史、收藏、主题和后台播放体验。

该顺序的目的是先稳定一个完整主链路，再逐步提升复杂能力，避免影视与音乐两条重链路同时全面铺开。

## 风险与边界

- 音乐插件兼容性是最大不确定性，可能需要针对具体插件做桥接兼容。
- 上游站点和第三方资源不稳定，后续维护会持续存在。
- 播放链路可能涉及重定向、鉴权、Referer、UA 和临时失效地址等复杂情况。
- 后台播放稳定性依赖服务生命周期、音频焦点与系统通知协同，需优先做真机验证。

## 验收标准

- 仓库中新增独立安卓目录 `flicker-app/`，不影响现有 Web 项目运行。
- 安卓工程具备多模块基础结构。
- 影视主链路可在安卓端独立完成浏览、搜索、详情和播放。
- 音乐主链路可在安卓端独立完成搜索、播放和歌词展示。
- 后台音乐播放、通知栏控制和锁屏控制可用。
- 不依赖现有 Node 服务即可完成 App 核心运行。
