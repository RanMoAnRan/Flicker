# Flicker 安卓 App 实施计划

## 目标

基于 [2026-03-22-flicker-android-app-design.md](/Users/ranmoanran/Jing/python-space/Flicker/docs/plans/2026-03-22-flicker-android-app-design.md) 的设计结论，为仓库中的 `flicker-app/` 安卓工程制定一份可执行的第一阶段实施计划。

本计划的目标不是一次性完成所有产品能力，而是建立一个可持续演进的安卓 App 基线，并优先打通最关键的业务链路。

## 计划范围

本计划覆盖：

- 新建独立安卓工程目录 `flicker-app/`
- 搭建多模块工程基线
- 优先完成影视主链路
- 再完成音乐基础链路
- 最后补齐设置、历史、收藏和播放体验能力

本计划暂不覆盖：

- 离线下载
- 账号体系
- 云同步
- 应用市场上架
- iOS 适配

## 实施原则

- 不破坏现有 Web 项目，安卓代码全部放在 `flicker-app/` 中
- 先建立清晰架构，再逐步补业务能力
- 优先完成单条主链路的稳定性，不追求初期功能大全
- 音乐插件能力单独攻坚，不与影视主链路同时大面积展开
- 每个阶段都必须有可验证的验收标准

## 目标目录结构

第一阶段工程目标结构如下：


## 阶段划分

实施分为六个阶段，按顺序推进。

### 阶段 0：工程基线

目标：

- 在 `flicker-app/` 下初始化 Android 工程
- 建立多模块结构
- 跑通最小可启动 App
- 确定依赖管理、模块依赖关系和基础主题

任务：

- 初始化 Gradle Kotlin DSL 工程
- 创建 `app`、`core-*`、`feature-*` 模块
- 接入 Compose、Navigation、ViewModel、Room、DataStore、Media3、Coil、OkHttp 等基础依赖
- 配置统一的 Android SDK、JDK、编译参数和版本管理
- 建立基础包名、应用名、主题、日志和 Result 封装
- 预留依赖注入入口

产出：

- `flicker-app/` 目录可独立构建
- App 可在模拟器或真机启动
- 首页可显示一个基础壳页面和模块导航占位

验收标准：

- 执行安卓构建命令能够成功产出调试包
- App 启动无崩溃
- 代码结构符合设计文档中的模块划分

### 阶段 1：通用基础能力

目标：

- 建立后续业务开发依赖的底座能力
- 把现有 Web 项目中的通用配置和状态管理映射到安卓

任务：

- 在 `core-common` 中建立错误模型、网络结果模型、分页结果模型
- 在 `core-network` 中建立 OkHttp 客户端、请求拦截器、日志与超时策略
- 在 `core-database` 中建立 Room 数据库、DAO、实体模型
- 在 `core-player` 中建立视频播放器和音频播放器基础封装
- 在 `feature-settings` 中完成主题、片源配置、当前片源偏好读写
- 将现有 `localStorage` 中的关键概念映射到 `DataStore` 与 `Room`

产出：

- 安卓端的基础数据层、网络层和存储层可独立使用
- 可以在设置页查看与修改基础配置

验收标准：

- 片源配置可本地保存和读取
- 主题设置能持久化
- 网络层支持基础请求和统一错误返回

### 阶段 2：影视主链路 MVP

目标：

- 优先打通安卓 App 最核心的影视使用链路

任务：

- 定义影视统一模型：`VideoSource`、`VideoCategory`、`VideoListItem`、`VideoDetail`、`Episode`、`PlayLine`
- 建立 `SourceAdapter` 机制，统一不同片源字段
- 完成首页分类浏览、推荐位和分页列表
- 完成影视搜索
- 完成详情页展示
- 完成线路和剧集选择
- 完成播放页接入 `Media3 ExoPlayer`
- 增加最近播放记录

产出：

- 安卓端能够完成影视浏览、搜索、详情和播放

验收标准：

- 首页能够基于当前片源加载分类和内容
- 搜索结果可正常展示并进入详情
- 详情页可展示影片信息、剧集和线路
- 至少一种主流片源能稳定完成播放

### 阶段 3：音乐基础链路 MVP

目标：

- 在不引入多插件聚合复杂度的前提下，先打通音乐搜索、播放和歌词

任务：

- 定义音乐统一模型：`MusicPlugin`、`Song`、`Album`、`Artist`、`LyricLine`、`MediaStream`
- 建立 `core-plugin` 的最小运行时骨架
- 先支持单插件加载与执行
- 实现关键词搜索
- 实现播放地址获取
- 实现歌词获取与时间轴渲染
- 在 `core-player` 中接入后台音乐播放、通知栏和锁屏控制

产出：

- 安卓端可完成单插件音乐搜索、播放、歌词展示和后台播放

验收标准：

- 至少一个已知稳定插件能完成搜索和播放
- 歌词可随播放进度同步展示
- 锁屏和通知栏控制正常可用

### 阶段 4：音乐聚合与播放队列

目标：

- 在音乐基础链路稳定后，再扩展多插件聚合能力

任务：

- 建立 `PluginManager`
- 支持多插件并发搜索
- 统一聚合结果排序与去重策略
- 建立播放队列和上下首逻辑
- 加入插件超时、异常隔离和兼容映射层
- 增加推荐插件配置能力

产出：

- 音乐模块具备接近当前 Web 版本的聚合能力

验收标准：

- 多插件搜索可返回聚合结果
- 单个插件失败不影响整体搜索结果
- 上一首、下一首和当前列表播放队列正常工作

### 阶段 5：体验补齐

目标：

- 把 MVP 从“能用”提升到“适合日常使用”

任务：

- 增加收藏和最近搜索
- 增加海报、歌词和插件目录缓存
- 完善弱网提示和重试交互
- 增加播放状态恢复
- 补齐设置页中的调试项和推荐插件控制
- 优化首页、详情页和播放页的加载状态

产出：

- 安卓端具备更完整的日常使用体验

验收标准：

- 收藏、历史、搜索记录正常保存
- 弱网或个别源失败时页面不出现全局白屏
- 应用重启后可恢复部分用户状态

### 阶段 6：稳定性与发布准备

目标：

- 为小范围分发和持续迭代做准备

任务：

- 补齐关键单元测试和集成测试
- 补齐核心 UI 测试
- 进行真机回归
- 优化崩溃日志和调试信息
- 产出 Debug 与 Release 构建说明

产出：

- 一版可在真机长期使用和分发的安装包

验收标准：

- 关键主流程通过真机回归
- 后台播放稳定
- 构建与安装流程清晰可复现

## 模块任务拆解

### `app`

- 应用入口
- 导航宿主
- 全局主题
- 模块装配

### `core-common`

- `Result` 和错误模型
- 通用工具
- 时间与格式化
- 日志封装

### `core-network`

- OkHttp 客户端
- 请求拦截器
- Header 注入
- 错误映射
- JSON 和文本解析基础能力

### `core-database`

- Room 数据库
- 片源配置表
- 历史记录表
- 收藏表
- 搜索历史表
- 缓存索引表

### `core-player`

- 视频播放器封装
- 音频播放器封装
- `MediaSessionService`
- 通知栏与锁屏控制
- 播放状态同步

### `core-plugin`

- JS 运行时接入
- 插件上下文管理
- 受控桥接接口
- 超时与异常隔离
- 插件兼容策略

### `feature-video`

- 首页
- 分类列表
- 搜索
- 详情
- 播放入口

### `feature-music`

- 音乐搜索
- 歌词展示
- 当前播放队列
- 推荐插件与聚合搜索

### `feature-settings`

- 片源管理
- 当前片源切换
- 主题切换
- 调试配置

## 关键依赖顺序

实施过程中按以下依赖关系推进：

1. `core-common`
2. `core-network` 与 `core-database`
3. `feature-settings`
4. `core-player`
5. `feature-video`
6. `core-plugin`
7. `feature-music`

原因：

- `feature-settings` 先落地后，影视和音乐都能直接复用片源与偏好配置。
- `feature-video` 的复杂度和不确定性低于音乐插件系统，适合作为首条业务主链路。
- `core-plugin` 稳住后，再进入音乐聚合能力，会更容易定位问题。

## 第一阶段建议任务清单

如果从下一步立即开始开发，我建议优先执行以下任务：

1. 创建 `flicker-app/` 工程骨架和多模块目录
2. 配置 Compose、Navigation、Room、DataStore、OkHttp、Media3
3. 建立基础主题、导航和空白首页
4. 实现设置页中的片源配置读取与保存
5. 建立影视统一模型和最小 `VideoRepository`
6. 先接通首页列表和搜索
7. 再接详情和视频播放

只有在影视主链路稳定后，再开始：

1. 接入 `QuickJS`
2. 建立单插件执行能力
3. 接通音乐搜索、播放和歌词

## 风险控制

针对本项目的主要风险，计划中的控制方式如下：

- 音乐插件兼容性：先从单插件、已验证插件入手，不一开始追求全量兼容
- 上游源不稳定：统一错误模型，页面按局部失败设计
- 播放异常复杂：将播放解析从页面中剥离到独立解析层
- 维护成本过高：从第一天开始按模块划分，不走单模块堆功能的路线

## 完成定义

当以下条件同时满足时，视为第一阶段 MVP 完成：

- `flicker-app/` 可独立构建并安装运行
- 影视主链路可完成浏览、搜索、详情和播放
- 音乐基础链路可完成搜索、播放和歌词展示
- 后台音乐播放与锁屏控制正常
- 设置页可管理基础片源与主题
- 关键路径经过至少一轮真机回归
